# 경기 상태별 분석 모드 개발

> 작성일: 2026-04-27
> 목적: 경기 전/진행 중/경기 후 상태에 따라 데이터 수집과 LLM 프롬프트를 다르게 적용하기 위한 개발 방향 정리.

---

## 문제 인식

현재 앱은 모든 경기를 같은 방식으로 처리함.

- 경기 전 경기: 선발, 팀 타선, 최근 흐름을 바탕으로 "매치업 예상 분석"을 생성함.
- 경기 후 경기: 이미 결과가 나왔는데도 여전히 "오늘 경기 예상" 프롬프트를 사용함.
- 과거 날짜 경기: 결과가 확정된 경기인데도 예측성 분석처럼 보일 수 있음.

즉, 현재 구조에서는 `2026-04-24`처럼 이미 종료된 경기를 봐도 앱은 "누가 유리한가", "예상 경기 성격" 같은 경기 전 관점으로 분석할 수 있음. 이 경우 데이터와 분석 목적이 어긋남.

---

## 핵심 인사이트

경기 분석 모드는 경기 상태에 따라 달라져야 함.

| 경기 상태 | 기준 | 앱이 해야 할 일 | 프롬프트 성격 |
|-----------|------|----------------|---------------|
| 경기 전 | 현재 시각이 경기 시작 전 | 선발/팀 지표/최근 흐름 기반 예상 분석 | Preview |
| 진행 중 | 경기 시작 후, 종료 전 | 실시간 스코어/이닝/투수 교체 등을 반영한 진행 상황 분석 | Live |
| 경기 후 | 경기 종료 후 | 실제 스코어, 승패, 선발 결과, 핵심 지표를 바탕으로 리뷰 리포트 | Recap |
| 과거 경기 | 조회 날짜가 과거 | 기본적으로 경기 후 리포트 | Recap |

예: 18:30 경기라면

- 18:30 전: "경기 전 예상 분석"
- 18:30 이후 종료 전: "진행 중 분석"
- 경기 종료 후: "경기 결과 리포트"

---

## 현재 코드의 한계

### 1. 경기 상태 모델이 없음

현재 `fetch_game_list()` 결과에는 다음 정보만 있음.

- `game_id`
- `date`
- `away`
- `home`
- `time`
- `away_pitcher`
- `home_pitcher`

하지만 아래 상태 정보가 없음.

- 경기 전인지
- 진행 중인지
- 종료됐는지
- 홈/원정 점수
- 승패 투수
- 경기장
- 실제 결과 기준 리포트에 필요한 이벤트/요약 데이터

---

### 2. 프롬프트가 하나뿐임

현재 `analysis/prompt.py`의 `build_prompt()`는 경기 전 예측성 분석을 기준으로 되어 있음.

현재 섹션:

- 유리한 팀
- 핵심 매치업 포인트
- 예상 경기 성격
- 주의 변수

경기 후에는 이 구조가 맞지 않음. 이미 결과가 확정됐으므로 다음과 같은 섹션이 필요함.

- 최종 결과
- 승부처
- 선발 투수 결과 평가
- 팀 타선/불펜/수비 요약
- 예상과 달랐던 점
- 다음 경기 참고 포인트

---

### 3. UI도 상태를 구분하지 않음

현재 앱 제목과 버튼, 분석 결과 섹션이 모두 "선발 매치업 분석" 관점임.

경기 후에는 사용자가 기대하는 정보가 다름.

- 경기 전: "누가 유리한가?"
- 경기 후: "왜 이겼나/졌나?"
- 과거 경기: "그 경기에서 무슨 일이 있었나?"

---

## 필요한 데이터 확장

### Game 모델 확장

`crawlers/types.py`의 `Game`에 아래 필드 추가를 검토함.

```python
class GameStatus:
    PREVIEW = "preview"
    LIVE = "live"
    FINAL = "final"


class Game(TypedDict):
    game_id: str
    date: str
    away: str
    home: str
    time: str
    status: str
    away_score: int | None
    home_score: int | None
    away_pitcher: str
    home_pitcher: str
    stadium: str
```

상태 판단에 필요한 원천:

- Schedule 페이지의 경기 상태 문구
- GameCenter 카드의 `game-cont` class
- `result_ck`, `start_ck`, `lineup_ck` 속성
- 스코어 영역
- 현재 시각과 경기 시작 시각

---

## 상태 판단 로직 제안

### 우선순위

상태 판단은 HTML에 있는 명시적 상태를 우선해야 함.

1. KBO HTML에서 `경기종료`, `result_ck=1`, 종료 class 확인
2. 진행 중 상태 문구 또는 스코어/이닝 정보 확인
3. 명시 상태가 부족하면 `date + time`과 현재 시각 비교

### 예시 로직

```python
def infer_game_status(game_date: str, game_time: str, html_status: str, result_ck: str) -> str:
    if result_ck == "1" or "경기종료" in html_status:
        return "final"

    start_dt = parse_kst_datetime(game_date, game_time)
    now = datetime.now(KST)

    if now < start_dt:
        return "preview"

    return "live"
```

주의:

- 한국 시간 기준으로 판단해야 함.
- 경기 취소/우천 지연/더블헤더 같은 예외 상태도 나중에 추가해야 함.

## 프롬프트 분리 제안

### 1. 경기 전 프롬프트

파일 예시:

- `analysis/prompts/preview.py`

목적:

- 선발 매치업
- 팀 타격/투구 지표
- 최근 5경기 흐름
- 유리한 팀과 변수

현재 `build_prompt()`는 이 모드로 유지 가능함.

---

### 2. 경기 후 프롬프트

파일 예시:

- `analysis/prompts/recap.py`

목적:

- 실제 경기 결과 설명
- 스코어 기반 리뷰
- 승패 요인
- 선발 투수 결과
- 타선/불펜/수비 요약

예상 출력 구조:

```markdown
## 1. 최종 결과
**LG 4-1 두산** — LG 승리

## 2. 승부처
(경기를 가른 핵심 장면 또는 지표)

## 3. 선발 투수 평가
(양 팀 선발의 실제 결과와 경기 영향)

## 4. 팀별 요약
(타선, 마운드, 흐름)

## 5. 다음 경기 참고 포인트
(다음 경기에서 이어질 변수)
```

---

### 3. 진행 중 프롬프트

파일 예시:

- `analysis/prompts/live.py`

목적:

- 현재 스코어와 이닝 기준 분석
- 남은 이닝 변수
- 불펜/대타/수비 교체 가능성

단, 진행 중 분석은 실시간 데이터 신뢰도가 중요하므로 P2 또는 P3 이후로 미루는 것이 좋음.

---

## 앱 흐름 개선안

현재:

```text
경기 선택
  -> pitcher_stats, lineup, recent_games 수집
  -> build_prompt()
  -> analyze_stream()
```

개선 후:

```text
경기 선택
  -> game.status 확인
  -> 공통 데이터 수집
  -> status별 추가 데이터 수집
  -> status별 prompt builder 선택
  -> analyze_stream()
```

예시:

```python
if game["status"] == "final":
    result_data = fetch_game_result(game["game_id"])
    prompt = build_recap_prompt(game, result_data, pitcher_stats, lineup)
elif game["status"] == "live":
    live_data = fetch_live_game(game["game_id"])
    prompt = build_live_prompt(game, live_data, pitcher_stats, lineup)
else:
    prompt = build_preview_prompt(game, pitcher_stats, lineup, recent_games)
```

## 우선 구현해야 하는 최소 버전

가장 먼저 필요한 최소 개선은 아래 3개임.

- [ ] `Game.status` 추가
- [ ] 종료 경기 스코어 파싱
- [ ] `preview`와 `recap` 프롬프트 분리

이 3개만 해도 과거 경기에서 예측 분석이 나오는 가장 큰 문제는 해결됨.

---

## 리스크와 주의점

- KBO HTML 구조가 경기 전/후뿐 아니라 취소, 우천 지연, 더블헤더에서 또 달라질 수 있음.
- 경기 상태 판단을 시간만으로 하면 틀릴 수 있음. 반드시 HTML 상태를 우선해야 함.
- 경기 후 리포트에 필요한 데이터가 부족하면 LLM이 추측할 위험이 있음. 프롬프트에서 "제공된 데이터 밖의 장면을 지어내지 말 것"을 강하게 제한해야 함.
- 과거 경기에서 선발 이름은 카드에서 파싱 가능하지만, 세부 투구 결과는 별도 데이터가 필요할 수 있음.

---

## 결론

현재 앱의 가장 큰 개념적 문제는 모든 경기를 "경기 전 예측"으로 처리한다는 점임.

앞으로는 `Game.status`를 중심에 두고,

- 경기 전: preview
- 진행 중: live
- 경기 후/과거: recap

으로 분석 모드를 분리해야 함.

이 변경은 단순 프롬프트 수정이 아니라 데이터 모델, 크롤러, UI, 프롬프트를 함께 바꾸는 작업임. 최소 구현은 `preview/recap` 두 모드 분리부터 시작하는 것이 적절함.
